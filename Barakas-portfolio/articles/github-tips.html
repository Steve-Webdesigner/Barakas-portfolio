Introduction

GitHub is the world's largest platform for version control and collaboration. For new developers, it can feel overwhelming—terms like repositories, branches, pull requests, and forks are unfamiliar. But mastering GitHub is essential for any developer. It's where code lives, where teams collaborate, and where you showcase your work to potential employers. In this guide, I'll walk through the GitHub concepts and workflows that I use every day.

Repositories

A repository (or "repo") is a project folder that contains all your project files and the revision history. Think of it as a database of your code that tracks every change you make. When starting a new project, the first step is creating a repository—either locally with git init or on GitHub itself.

Best practices for repositories:

  • Use a clear, descriptive name (e.g., internhub-kenya, personal-portfolio)
  • Always include a README.md that explains what the project does
  • Add a .gitignore file to exclude unnecessary files (node_modules, .env, etc.)
  • Choose the right license for your project

Commits

Commits are snapshots of your project at specific points in time. Each commit records what changed, who made the change, and when. Writing good commit messages is a skill that pays dividends when you need to understand why a change was made months later.

I follow the conventional commits format: type(scope): description. Examples include feat: add search functionality, fix: resolve mobile nav bug, and docs: update README. This makes the commit history readable and searchable.

Branches

Branches allow you to work on different features or fixes simultaneously without affecting the main codebase. The default branch is usually main or master. When starting work on a new feature, I create a new branch off main, do my work there, then merge it back.

My branching strategy:

  • main — Production-ready code
  • dev — Development branch for testing
  • feature/feature-name — Individual feature branches
  • fix/bug-description — Bug fix branches

Pull Requests

A pull request (PR) is how you propose changes to a repository. It's a request to merge changes from one branch into another. PRs are the heart of collaboration on GitHub—they enable code review, discussion, and quality control.

When creating a PR, I include a description of what the changes do, why they were made, and any relevant screenshots or links. I also reference any related issues using GitHub's issue linking syntax (#issue-number).

GitHub Pages

GitHub Pages is a free hosting service that turns your repositories into live websites. It's perfect for hosting portfolio sites, project demos, and documentation. Setting it up is simple: go to your repository settings, enable GitHub Pages, and select the branch to deploy.

I use GitHub Pages to host project documentation and demo versions of my frontend projects. It's an excellent way to share your work without paying for hosting.

Portfolio Deployment

Your GitHub profile is your developer resume. Keep it organized and professional. Here are my tips:

  • Pin your best repositories to the top of your profile
  • Write great READMEs for every project with screenshots, live links, and setup instructions
  • Contribute to open source — even small contributions build your reputation
  • Keep your activity graph green by committing regularly
  • Use GitHub Actions for continuous integration and deployment

Conclusion

GitHub is more than just a code hosting platform—it's a portfolio, a collaboration tool, and a learning resource. By mastering repositories, commits, branches, pull requests, GitHub Pages, and portfolio best practices, you'll set yourself up for success as a developer. Start by creating your first repository today, and commit to using Git and GitHub in every project going forward.
